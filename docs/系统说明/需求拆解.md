# 后稷灌溉 AI 运营系统需求拆解

版本：2026-03-26  
状态：active  
适用范围：Phase 1

## 1. 拆解原则

本拆解按“大颗粒业务能力”组织，不按个人分工拆，不按零散页面拆。

目标是形成：

- 可冻结的数据骨架
- 可对齐的前后端边界
- 可联调的阶段目标
- 可验收的业务里程碑

## 2. 一级能力拆解

Phase 1 建议拆成 7 个一级工作流：

1. 项目与区块主数据
2. 资产、设备与空间录入
3. 管网模型与 DWG 配置
4. 自动调度与驾驶舱
5. 会话、运维与告警闭环
6. 电量计量与成本核算
7. 权限、审计与 UAT

## 3. 各工作流说明

### 3.1 项目与区块主数据

#### 目标

建立“区域 -> 项目 -> 区块”的稳定主线。

#### 范围

- 项目主数据
- 区块主数据
- 项目规则
- 区块统计
- 项目与区块权限边界

#### 验收结果

- 项目可作为经营主体独立存在
- 区块可作为调度与核算单元独立存在
- 项目中的区块数量由系统自动统计

### 3.2 资产、设备与空间录入

#### 目标

把资产、设备、现场点位和地图空间信息打通。

#### 范围

- 资产主数据
- 设备主数据
- 一物一码
- 现场录入
- 经纬度 / 海拔 / 地址获取
- 台区资产
- 电表设备

#### 验收结果

- 资产和设备可绑定项目、区块、模型节点
- 现场到点录入后可直接进入正式台账或待校核状态

### 3.3 管网模型与 DWG 配置

#### 目标

建立项目级可版本化管网模型。

#### 范围

- 管网模型版本
- 节点 / 管线 / 泵站 / 阀门 / 出水口
- DWG 导入
- 模型草稿校核
- 发布与回滚

#### 验收结果

- 项目可导入 DWG 并生成模型草稿
- 模型能发布为正式版本供驾驶舱和求解服务使用

### 3.4 自动调度与驾驶舱

#### 目标

建立默认自动调度、可解释、可预警、可回放的项目 / 区块驾驶舱。

#### 范围

- solver service
- 自动选泵 / 选路 / 阀门动作序列
- 项目驾驶舱
- 区块驾驶舱
- 调度解释
- 历史回放
- UAT 仿真

#### 验收结果

- 正常场景可自动求解
- 驾驶舱能展示项目态势、区块态势和预警解释

### 3.5 会话、运维与告警闭环

#### 目标

把灌溉执行、异常处理和工单巡检打通。

#### 范围

- 会话
- 订单
- 告警
- 工单
- 巡检
- 维护团队
- 自动调度异常联动

#### 验收结果

- 自动调度异常能进入告警和工单链路
- 维护团队可承接项目和资产级运维任务

### 3.6 电量计量与成本核算

#### 目标

让电量计量、项目成本和区块成本形成正式口径。

#### 范围

- 电力计量点
- 周期抄表
- 电价方案
- 成本归集
- 区块成本
- 项目成本

#### 验收结果

- 能按计量点 -> 区块 -> 项目形成核算链
- 财务能按周期核算主要电力成本

### 3.7 权限、审计与 UAT

#### 目标

在大规模设备与项目场景下保持权限可控、审计可追、UAT 可验证。

#### 范围

- 功能权限
- 数据权限
- 高风险动作审计
- UAT 场景
- 测试数据清理

#### 验收结果

- 数据权限主颗粒度落到项目 / 区块
- 高风险动作可追溯
- UAT 能验证自动调度和成本核算关键场景

## 4. 开发任务与进度建议

建议按 5 个大阶段推进，每阶段颗粒度保持在“可形成联调成果”级别，而不是碎页面级别。

### 阶段 A：骨架冻结

状态建议：最高优先级  
目标：先把不会轻易推翻的骨架定下来。

包括：

- 核心表和主键
- 核心编码规则
- 区块对象
- 电力计量点对象
- 权限骨架
- solver service 接口边界

### 阶段 B：项目配置底座

状态建议：紧接 A 启动

包括：

- 项目与区块
- 资产与设备
- 一物一码
- 搜索能力
- 现场录入
- DWG 导入草稿链

### 阶段 C：调度与驾驶舱

状态建议：B 的正式模型可用后启动

包括：

- 项目驾驶舱
- 区块驾驶舱
- 自动调度
- 调度解释
- UAT 仿真

### 阶段 D：运维与财务闭环

状态建议：C 跑通后启动

包括：

- 告警 / 工单 / 巡检
- 维护团队联动
- 电量计量
- 成本归集
- 项目经营看板

### 阶段 E：联调收口与权限收口

状态建议：首发前最后阶段

包括：

- 全局权限
- 高风险动作审计
- 数据权限收口
- UAT 全量验证
- 测试数据清理

## 5. 当前推荐进度判断

结合当前方向，推荐的项目推进口径是：

1. 当前先把阶段 A 做实
2. 阶段 B 与阶段 C 可以部分并行，但前提是骨架先冻结
3. 阶段 D 在自动调度主线跑通后集中推进
4. 阶段 E 必须单独收口，不能边上线边补

## 6. 哪些东西适合先冻结，哪些不适合

### 6.1 适合现在先冻结

- 项目
- 区块
- 资产
- 设备
- 电力计量点
- 管网模型版本
- 会话 / 订单主线
- 告警 / 工单主线
- 功能权限和数据权限骨架

### 6.2 不适合现在冻结得太死

- 复杂调度算法细节
- 专变分摊细节
- 高级优化器参数
- 深度 AI 推理策略

## 7. 一句话结论

当前最优推进方式不是零碎修页面，而是先把“项目 / 区块 / 资产 / 设备 / 计量点 / 模型 / 权限 / 求解边界”这套骨架冻结，再围绕它做配置、驾驶舱、运维和财务闭环。
