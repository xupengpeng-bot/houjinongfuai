# 灌溉启动与刷卡扫码专家经验库

状态：active

## 目标

本文件用于沉淀灌溉现场“刷卡启动、扫码启动、停止、结算、留痕”的正式业务口径。

它不是一次性说明，而是持续维护的专家经验库：

- 每次涉及启动、停止、结算、补发、设备联调、刷卡扫码交互的开发，都必须先对照本文件。
- 每次发现历史规则有问题，必须直接修正文档，不能只改代码不改规则。
- 文档中的规则优先级高于临时口头约定，高于“为了方便实现”的平台侧折中方案。

## 核心原则

### 1. 启动必须是同步结果

- 刷卡启动、扫码启动都必须做成同步确认链路。
- 平台只有在设备已经明确启动成功后，才能返回“启动成功”。
- 设备离线、指令未送达、超时、`NACK`、流程中任一步失败，都必须直接返回“启动失败”。
- 不允许把启动结果留成“稍后可能成功”的悬空状态。

### 2. 平台禁止补发启动命令

- 所有启动类命令禁止平台自动重试。
- 设备重连后，禁止把历史启动命令重新激活。
- 人工侧也禁止对启动类命令做“补发”“重发”“补偿启动”。
- 原因：农户现场刷一下没出水就会离开，如果平台后面又自动开起来，会直接造成严重现场风险。

### 3. 刷卡去抖必须放在前端设备端

- 刷卡去抖不能放在平台。
- 农户存在“刚刷卡开始，马上又刷卡结束”的真实现场习惯。
- 如果平台做去抖，会把用户真实的第二次操作吞掉，出现“已经结束了，但订单还在走”的问题。
- 设备端可以根据硬件读卡器特性做短时去抖，但平台必须尊重每一次用户操作意图。

### 4. 刷卡流水只做事件留痕，不负责业务裁决

- `card_swipe_event` 的职责是记录“发生过什么刷卡事件”，不是替代订单流程。
- 所有刷卡事件都必须有留痕，包括：
- 缺少卡号
- 无效卡
- 启动成功
- 启动失败
- 停止成功
- 停止失败
- 设备本地拒绝
- 平台拒绝
- 连接中断
- 业务层可以失败，但刷卡流水不能缺失。

### 5. 订单金额在结算后必须冻结

- 会话结束并结算后，订单金额必须停止增长。
- 已结算订单禁止继续根据后续影子数据、设备上报、查询补偿去刷新金额。
- “结束”和“结算”是金额冻结点，不允许出现订单已经结束但金额还在继续走的情况。
- 停机请求一旦被平台受理，会话即使还在等待设备停机确认，订单金额也必须先按“停机受理时刻”的指标快照冻结。
- `stopping` 期间禁止继续按当前时钟时长、实时累计流量、实时累计电量去推高订单金额。
- 只有在设备最终上报停机结果并进入最终结算时，才允许基于最终停机事实重新核定应结金额。

## 具体业务规则

### A. 刷卡启动

- 平台接收到刷卡启动后，先记录刷卡事件，再进入业务判断。
- 如果设备不能立刻确认启动成功，则本次启动直接失败。
- 启动失败后：
- 会话关闭
- 订单关闭
- 预付/锁定资金立刻退款或解锁
- 返回前端明确失败结果

### B. 刷卡停止

- 同一张启动卡应具备关闭本次订单的优先权。
- 农户在同一目标设备上再次刷卡，平台应优先按“可能是停止意图”理解，而不是简单拦截为重复启动。
- 停止指令可以等待设备确认，但不能因为平台的启动去抖或补发机制把流程搅乱。

### C. 扫码支付后启动

- 支付成功不等于设备已经启动成功。
- 扫码支付回调后，平台必须立即走同步启动确认。
- 如果设备没有成功启动，本次交易必须按启动失败处理，不能保留“已支付、待补发启动”的中间态。

### D. 超时与离线

- 启动链路中只要出现设备离线、发送失败、确认超时，都按失败处理。
- 超时失败后不能等待设备后续恢复再继续执行原启动命令。
- 恢复连接只能用于后续新的、明确发起的启动请求。

## 开发约束

- 任何“为了提高成功率”而引入的平台侧自动补发、重试、断线恢复重投，默认视为高风险设计。
- 任何“为了减少重复点击/重复刷卡”而在平台层吞事件的做法，默认视为高风险设计。
- 涉及订单、会话、支付、设备指令、刷卡流水的改动，必须从现场操作习惯反推，而不能只从服务端状态机出发。

## 持续更新要求

- 后续开发只要触碰以下模块，必须同步更新本经验库：
- 启动/停止状态机
- 设备网关指令投递
- 刷卡扫码入口
- 订单结算
- 钱包锁定与退款
- 支付回调
- 刷卡事件留痕

- 更新原则：
- 新现场问题先归因到规则，再落代码。
- 发现旧规则不对时，直接改规则正文，不追加“补丁式附录”。
- 文档必须始终保持“当前正式口径”。
