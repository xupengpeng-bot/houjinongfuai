# 后稷灌溉 AI 运营系统业务总览简版

版本：2026-03-26  
状态：active  
用途：供 PM、Cursor、Lovable、评审者快速对齐系统主线业务，不替代正式需求文档。

## 1. 这套系统到底在做什么

这不是一个泛农业大平台的首发版，而是一套围绕灌溉场景的运营系统。

Phase 1 只有一个核心目标：

跑通“项目经营闭环 + 现场运维闭环 + 日志审计闭环”，并在此基础上接入项目级区块驾驶舱和自动调度能力。

也就是说，系统必须同时让四件事成立：

1. 农户能真实使用
2. 项目能真实经营
3. 运维和财务能真实追责与核算
4. 自动调度能在后台稳定运行，驾驶舱能看懂、能预警、能回放

## 2. 系统主线结构

系统围绕以下对象运行：

- 区域
- 项目
- 区块
- 站点
- 资产
- 设备
- 电力计量点
- 管网模型
- 运行会话
- 告警 / 工单 / 巡检
- 维护团队
- 驾驶舱

其中最关键的主线是：

```text
区域
  └─ 项目
      ├─ 区块
      │   ├─ 管网模型
      │   ├─ 出水口
      │   ├─ 泵站 / 阀门 / 节点 / 管线
      │   ├─ 传感器 / 控制器 / 电表
      │   ├─ 电力计量点
      │   └─ 调度与预警
      ├─ 资产
      ├─ 设备
      ├─ 会话 / 订单
      ├─ 告警 / 工单
      ├─ 成本核算
      └─ 调度驾驶舱
```

## 3. 最重要的业务原则

### 3.1 项目是经营主体

收入、成本、毛利、维护归属、运营统计都以项目为主口径。

### 3.2 区块是调度与运营单元

驾驶舱默认打开的是“项目下某个区块”的视角，而不是整个项目的一锅端大屏。

### 3.3 自动调度为主，人工处理异常

系统从一开始就不依赖大量调度员实时管理。  
默认流程是：

- 后端自动选泵
- 自动选路
- 自动联动
- 自动预警
- 驾驶舱负责展示、解释和异常干预

### 3.4 前端不承载业务真相

前端只消费后端 contract：

- 不在前端保留正式调度真相
- 不在前端做正式求解
- 不在前端做正式控制决策
- 不让前端直连第三方业务/地理真相服务

### 3.5 资产与设备分层

- 资产：工程对象、维修对象、财务对象
- 设备：控制对象、采集对象、计量对象

典型例子：

- 变压器台区：资产
- 国网电表：设备
- 电力计量点：核算对象

### 3.6 一物一码

项目、区块、资产、设备、计量点、模型版本、工单、会话等正式对象都应具备稳定唯一的业务编码，系统自动生成，不允许用户手填。

## 4. 当前已经冻结的关键产品口径

### 4.1 配置原则

- 能自动生成的，不让用户手填
- 能下拉选择的，不让用户自由输入
- 数量大时必须支持搜索
- 统计类字段只读展示
- 经纬度、海拔、面积、中心点优先自动获取或自动计算

### 4.2 区域与项目

- 项目当前只保留一个区域选择流
- 不再重复维护第二套项目物理行政区选择器
- 项目中的区块数量由系统自动统计，不允许人工填写

### 4.3 资产位置

- 资产位置先继承项目和区块上下文
- 再在项目范围内做搜索、选点和微调
- 现场录入时应支持自动获取经纬度、海拔、地址描述

### 4.4 驾驶舱

- 驾驶舱默认按“项目 > 区块”打开
- 项目总览驾驶舱是上层概览，不替代区块驾驶舱

### 4.5 DWG 与现场录入

- DWG 导入用于初始化项目管网模型
- 现场运维到点录入用于校正和补全空间真相
- DWG 草稿和现场录入结果都必须回落到正式模型与台账

## 5. 这个系统怎么服务不同角色

### 5.1 农户

只看到：

- 扫码 / 刷卡
- 开始灌溉
- 停止灌溉
- 当前状态
- 历史记录
- 账户 / 余额

### 5.2 运营和运维

看到：

- 项目态势
- 区块驾驶舱
- 自动调度结果
- 运行监控
- 预警解释
- 工单 / 巡检 / 维护团队

### 5.3 财务

看到：

- 周期抄表
- 电量计量
- 成本归集
- 区块成本分析
- 项目成本分析

## 6. waterflow-control 的定位

接入后的定位不是农户首页，而是：

- 项目级调度驾驶舱
- 区块级数字孪生视图
- 自动调度结果解释与预警入口
- UAT 仿真平台

其中求解能力建议迁到后端专门 solver service，前端保留：

- 地图与管网可视化
- 调度结果展示
- 告警与风险解释
- 配置交互
- UAT 仿真界面

## 7. 首发必须做到什么程度

1. 项目、区块、站点、资产、设备、计量点和维护团队主线跑通
2. 农户可正常发起和结束灌溉
3. 后端能自动调度并产出可解释结果
4. 驾驶舱能看项目态势、区块态势、预警和历史
5. 财务能按计量点 / 区块 / 项目做成本归集
6. 权限和审计能管住高风险动作

## 8. 给协作方的最短理解

### 8.1 给后端

- 主后端是唯一业务真相
- solver service 负责自动求解
- 项目与区块是核心业务上下文
- 数据权限主颗粒度优先到项目 / 区块

### 8.2 给前端

- 驾驶舱是可视化与解释层，不是业务真相层
- 配置页要遵守自动生成、下拉优先、搜索优先、一物一码
- 所有地图与定位能力都必须受后端 contract 管控

### 8.3 给 PM / 评审者

优先判断一项需求是否服务于：

1. 项目经营闭环
2. 区块级自动调度与预警
3. 运维与财务闭环
4. 审计与权限闭环
