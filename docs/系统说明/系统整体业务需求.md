# 后稷灌溉 AI 运营系统整体业务需求

版本：2026-03-26  
状态：active  
适用范围：Phase 1

## 1. 文档目标

本文件用于定义系统的正式业务需求，作为后续：

- 数据骨架冻结
- 前后端联调
- 驾驶舱接入
- 自动调度建设
- UAT 验收

的统一业务真相。

本版已经融合以下新确认内容：

1. 主前端（`lovable-working`，含 `src/features/waterflow/` 等）提供项目级调度驾驶舱能力（历史上由独立 `waterflow-control` 合并而来，sidecar 目录已不再置于工作区根）
2. 驾驶舱默认打开“项目下某个区块”
3. 自动调度为主，人工主要处理异常
4. 资产/设备录入与地图、DWG、驾驶舱统一规划
5. 变压器台区归资产，国网电表归设备，新增电力计量点做成本核算
6. 一物一码、自动生成、下拉优先、搜索优先、统计只读
7. 前期数据库和权限骨架优先冻结，便于联调

## 2. 系统定位

系统 Phase 1 不是做一个大而全的农业平台，而是做成一套可运营、可调度、可核算、可运维、可审计的灌溉运营系统。

系统应同时支撑四条主线：

1. 项目经营主线
2. 现场运维主线
3. 自动调度与预警主线
4. 日志审计与权限主线

## 3. 核心对象与统一结构

### 3.1 统一业务结构

```text
区域
  └─ 项目
      ├─ 区块
      │   ├─ 管网模型
      │   ├─ 出水口
      │   ├─ 泵站 / 阀门 / 管线 / 节点
      │   ├─ 设备
      │   ├─ 电力计量点
      │   ├─ 调度参数
      │   └─ 预警规则
      ├─ 站点
      ├─ 资产
      ├─ 设备
      ├─ 维护团队
      ├─ 会话 / 订单
      ├─ 告警 / 工单 / 巡检
      ├─ 成本核算
      └─ 调度驾驶舱
```

### 3.2 对象角色定义

- 区域：治理和归属边界
- 项目：经营主体
- 区块：项目下可独立调度、独立统计、独立核算的最小业务单元
- 站点：现场组织和挂接骨架
- 资产：工程对象、台账对象、维修对象、财务对象
- 设备：控制对象、采集对象、计量对象
- 电力计量点：能源核算对象
- 驾驶舱：区块级运行态势、预警、调度结果和 UAT 仿真入口

## 4. 核心业务原则

### 4.1 项目是经营主体

收入、成本、毛利、维护归属、经营分析最终都以项目为主口径。

### 4.2 区块是调度与核算单元

区块不是装饰性对象，而是：

- 驾驶舱默认打开对象
- 自动调度默认求解对象
- 成本归集与统计对象

### 4.3 自动调度为主

系统默认不依赖大量调度员实时管理。  
正式运行原则为：

- 正常场景自动求解、自动执行
- 中风险场景自动求解并提示确认
- 高风险场景转入人工复核

### 4.4 后端是唯一业务真相

正式求解、控制、会话、计费、工单、预警、审计均由后端统一负责。

前端不承担：

- 正式求解真相
- 正式控制决策
- 正式业务状态机
- 正式权限判断

### 4.5 资产和设备必须分层

典型归类：

- 变压器台区：资产
- 国网电表：设备
- 电力计量点：核算对象

### 4.6 一物一码

项目、区块、资产、设备、计量点、模型版本、工单、告警、订单、会话等正式对象必须具备稳定唯一编码，系统自动生成，不允许用户手填。

## 5. 项目管理需求

项目管理用于建立经营主体，不承载复杂模型编辑。

### 5.1 项目管理必须支持

- 项目名称
- 项目编码（系统自动生成）
- 所属区域
- 项目负责人
- 联系电话
- 运营方
- 默认维护团队
- 项目状态
- 当前模型版本
- 调度模式

### 5.2 项目管理中的只读统计

项目页允许展示但不允许人工填写：

- 区块数量
- 资产数量
- 设备数量
- 出水口数量
- 活跃告警数

这些字段统一由系统自动统计。

## 6. 项目配置需求

项目配置是把一个项目从“有主数据”变成“可调度、可核算、可驾驶舱运行”的关键层。

### 6.1 区块配置

区块必须成为正式业务对象，支持：

- 区块名称
- 区块编码（自动生成）
- 所属项目
- 区块边界
- 区块中心点（自动计算，允许校正）
- 区块面积（自动计算，允许校核）
- 优先级
- 默认供水策略
- 默认电力计量点
- 状态

#### 中心点用途

- 驾驶舱默认打开位置
- 地图搜索默认聚焦
- 现场录入空间推荐
- 邻近资产 / 设备 / 节点绑定推荐

#### 面积用途

- 区块规模统计
- 单位面积灌溉量
- 单位面积耗电
- 单位面积成本
- 需求预测基线

### 6.2 管网模型配置

每个项目必须支持管网模型版本化管理。

模型至少包含：

- 节点
- 管线
- 泵站
- 阀门
- 出水口
- 传感器
- 连接关系
- 模型版本状态

### 6.3 DWG 导入

DWG 导入用于初始化模型，不直接成为运行真相。

标准流程：

1. 上传 DWG
2. 解析图层和对象
3. 生成模型草稿
4. 绑定区块、资产、设备、计量点
5. 人工校核
6. 发布模型版本

### 6.4 资产配置

资产配置必须同时服务：

- 台账管理
- 驾驶舱映射
- 运维闭环
- 财务核算

资产至少支持：

- 泵站
- 泵组
- 阀门 / 阀组
- 管道
- 节点
- 出水口
- 井房
- 变压器台区
- 配电箱 / 供电单元

### 6.5 设备配置

设备配置必须支持：

- 设备编码自动生成
- 设备类型下拉选择
- 所属项目 / 区块 / 资产绑定
- 所属模型节点绑定
- 所属计量点绑定
- 通讯标识
- 在线状态

设备重点包括：

- 控制器
- 传感器
- 网关
- 国网电表 / 自有计量表

### 6.6 电力计量点配置

电力计量点是后续财务核算的正式对象。

必须支持：

- 计量点名称
- 计量点编码（自动生成）
- 所属项目
- 所属区块
- 所属台区资产
- 主电表设备
- 子电表设备
- 电价方案
- 分摊规则
- 是否参与成本核算

### 6.7 调度与预警配置

项目和区块级都应支持：

- 自动调度策略
- 求解参数
- 风险阈值
- 自动执行条件
- 预警规则
- 电量异常规则

## 7. 现场录入需求

系统必须支持运维人员到点录入，不能只靠办公室纯手工维护。

### 7.1 现场录入对象

- 出水口
- 阀门
- 泵站
- 传感器
- 节点
- 管线端点
- 变压器台区
- 电表
- 电力计量点

### 7.2 现场录入时自动获取

- 经纬度
- 海拔
- 地址描述
- 现场照片
- 所属区块候选
- 附近模型节点候选

### 7.3 录入状态

- 待校核
- 已校核
- 已发布

## 8. 运营与驾驶舱需求

### 8.1 驾驶舱定位

主前端调度/管网能力接入后的正式定位是：

- 项目级调度驾驶舱
- 区块级数字孪生视图
- 自动调度结果解释层
- UAT 仿真平台

### 8.2 驾驶舱打开方式

默认路径：

1. 先进入项目
2. 再进入区块
3. 打开区块驾驶舱

同时允许项目总览驾驶舱作为上层概览。

### 8.3 驾驶舱必须支持

- 项目态势
- 区块运行态势
- 管网地图
- 自动调度结果
- 压力 / 流量 / 风险可视化
- 预警解释
- 历史回放
- UAT 仿真

## 9. 自动调度与 solver service 需求

### 9.1 设计原则

正式求解必须迁到后端专门 solver service。

### 9.2 solver service 职责

- 装载项目和区块模型
- 自动选泵
- 自动选路
- 自动生成阀门动作序列
- 风险评估
- 能耗预测
- 预警预判
- 结果解释

### 9.3 主后端职责

- 权限
- 资产 / 设备 / 区块 / 计量点真相
- 会话和订单
- 控制命令
- 审计
- 告警与工单联动

### 9.4 UAT 职责

UAT 要重点验证：

- 正常灌溉
- 资源冲突
- 泵故障
- 阀门失败
- 压力不足
- 无流量
- 电量异常
- 成本归集正确性

## 10. 成本核算需求

### 10.1 核算主线

每个核算周期必须支持：

1. 拉取电力计量点读数
2. 关联电表设备
3. 关联区块 / 台区 / 泵站 / 运行会话
4. 计算周期耗电
5. 形成区块成本和项目成本

### 10.2 财务视角

财务应至少能看到：

- 周期抄表
- 电量计量
- 区块成本
- 项目成本
- 收入 / 成本 / 毛利

## 11. 全局权限需求

### 11.1 功能权限

功能权限至少覆盖：

- 一级菜单
- 二级菜单
- 页面
- 按钮
- 接口
- 高风险动作

### 11.2 数据权限建议

推荐默认主颗粒度：

- 项目级
- 区块级

不推荐把设备级作为全局默认数据权限主颗粒度，因为：

1. 设备量可能非常大
2. 管理成本高
3. 查询和权限计算复杂度高
4. 大多数角色并不需要精确到单设备范围授权

### 11.3 推荐的权限层次

#### 默认数据权限

- 平台级
- 区域级
- 项目级
- 区块级

#### 例外细粒度权限

对于少量高风险角色，可追加：

- 资产级临时授权
- 设备级临时授权

这类权限不建议做成系统默认主模型，而应作为例外覆盖。

### 11.4 高风险动作

必须独立授权和审计：

- 强制停泵 / 停阀
- 手工改调度
- 改动求解参数
- 改动模型版本
- 批量导入覆盖
- 退款 / 冲正
- 导出敏感明细

## 12. 配置体验要求

配置体验必须遵守以下规则：

1. 能自动生成的，不让人手填
2. 能下拉选择的，不让人自由输入
3. 数据量大时必须支持搜索
4. 统计字段只读展示
5. 编码创建后不可随意修改
6. 所有用户侧枚举必须中文化

## 13. 首发上线门槛

首发前至少要满足：

1. 项目、区块、资产、设备、计量点、维护团队主线打通
2. 农户端灌溉主链路可用
3. 自动调度可运行并可解释
4. 驾驶舱能看项目和区块态势
5. DWG 导入和现场录入至少有一条主线可用
6. 成本核算能按计量点 -> 区块 -> 项目归集
7. 功能权限和数据权限主骨架已落地

## 14. 一句话结论

Phase 1 的本质不是把所有功能都做完，而是先把“项目经营 + 区块配置 + 自动调度 + 运维闭环 + 成本核算 + 权限审计”这条主线统一起来，形成一个既能真实运营、又能持续扩展的灌溉数字化底座。
