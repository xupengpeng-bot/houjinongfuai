# 骨架冻结与开发分期

版本：2026-03-26  
状态：active  
适用范围：Phase 1

## 1. 目标

本文件用于回答两个关键问题：

1. 前期数据库、表、字段等骨架性的东西能不能先确定下来  
   结论：**可以，而且建议尽快冻结主骨架**
2. 开发任务和进度如何划分，才能让前后端联调清晰  
   结论：**按大颗粒能力域分期推进**

## 2. 为什么建议先冻结骨架

不先冻结骨架，后续会出现四类问题：

1. 前端页面和后端 contract 同时漂移
2. DWG 导入、现场录入、驾驶舱、会话、核算各说各话
3. 权限和数据范围收口困难
4. UAT 很难验证“系统到底按什么对象模型在运行”

所以建议 Phase 1 先把以下内容固定下来：

- 核心对象
- 核心关系
- 核心编码
- 核心权限颗粒度
- solver service 边界

## 5. 一物一码建议

推荐所有正式对象采用“双层标识”：

1. 内部主键
   - 稳定，不面向普通用户
2. 业务编码
   - 可读、可搜索、可打印、可贴码

建议系统自动生成，不允许用户手填。

适用对象至少包括：

- 项目
- 区块
- 资产
- 设备
- 电力计量点
- 模型版本
- 工单
- 告警
- 会话 / 订单

## 6. 权限策略建议

## 6.1 功能权限

功能权限建议控制到：

- 一级菜单
- 二级菜单
- 页面
- 按钮
- 接口
- 高风险动作

## 6.2 数据权限

### 推荐默认主颗粒度

- 平台级
- 区域级
- 项目级
- 区块级

### 不推荐默认用设备级作为主颗粒度

原因：

1. 设备数量可能极大
2. 权限配置成本高
3. 查询和缓存复杂度高
4. 大多数角色并不需要设备级独立授权

### 推荐做法

默认数据权限走项目 / 区块级。  
对于极少量高风险角色，再用“例外覆盖”的方式授权到：

- 资产级
- 设备级

这样兼顾：

- 大规模设备场景下的效率
- 管理复杂性的可控

## 7. 前后端联调边界建议

建议在骨架冻结后，联调统一遵守：

1. 表和对象名先固定
2. 编码规则先固定
3. 主关系先固定
4. API 响应壳先固定
5. 前端不假设后端会继续大改对象模型

## 8. 开发分期建议

### 8.1 第 1 期：骨架冻结期

交付重点：

- 核心表
- 核心关系
- 编码规则
- 权限骨架
- solver service 接口骨架

### 8.2 第 2 期：配置底座期

交付重点：

- 项目
- 区块
- 资产
- 设备
- 电力计量点
- 搜索与自动生成
- 现场录入
- DWG 草稿导入

### 8.3 第 3 期：驾驶舱期

交付重点：

- 项目驾驶舱
- 区块驾驶舱
- 自动调度结果
- 风险解释
- 历史回放
- UAT 仿真

### 8.4 第 4 期：闭环期

交付重点：

- 告警 / 工单 / 巡检
- 维护团队联动
- 电量计量
- 成本归集
- 项目经营分析

### 8.5 第 5 期：首发收口期

交付重点：

- 全局权限收口
- 高风险动作审计
- UAT 收口
- 测试数据清理
- 上线门槛确认

## 9. 当前推荐结论

1. 前期数据库和字段骨架**应该先确定死**
2. 但冻结的是主骨架，不是所有高级算法细节
3. 数据权限主颗粒度推荐到项目 / 区块
4. 设备级权限只做例外能力，不做全局默认主模型
5. 这样最适合大规模设备应用和后续管理复杂度平衡
