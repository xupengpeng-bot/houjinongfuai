# 控制器角色与硬件 SKU 规划 v1

## 1. 结论先行

如果从业务长期可扩展、硬件 SKU 收敛、嵌入式实现可控这三个目标一起看，我建议这样定：

- 业务上的控制器角色：`5` 类核心角色
- 实际硬件主 SKU：`4` 类主 SKU
- 其中真正应该优先量产的主控板：`2` 类
- 其余做成站级网关和扩展板，不要每种业务都单独做一块板

推荐口径：

1. `站级网关控制器`
2. `通用水源控制器`
3. `阀控器（当前按单路）`
4. `采集监测控制器`
5. `远程 IO / 扩展控制器`

但在硬件上，不要做 5 套完全不同的产品。建议收敛成：

1. `H1 站级边缘网关`
2. `H2 通用 RTU 控制器`
3. `H3 单路阀控终端`
4. `H4 采集 / 远程 IO 扩展板`

可选附属件：

- `H5 操作面板 / HMI`

## 2. 为什么要分“角色”和“SKU”

业务上，系统需要知道：

- 这是不是机井主控
- 这是不是泵站主控
- 这是不是阀控器
- 这是不是监测采集器
- 这是不是站级网关

但硬件上，不应该为了这些业务名字，每种都做一套板。

更合理的做法是：

- 平台层看 `controller_role`
- 硬件层看 `hardware_sku`
- 一个 `hardware_sku` 可以通过不同固件模板，承担多个 `controller_role`

## 3. 业务上建议冻结的控制器角色

### 3.1 C1 站级网关控制器

用途：

- 站级或地块级统一接入
- 承担 4G / 以太网 / VPN / OTA / 本地缓存
- 聚合下挂 RTU、阀控板、采集板
- 对上只暴露一个主通信身份

适用场景：

- 泵站
- 大型机井群
- 水肥站
- 一个点位下挂多块子控制板

是否必须独立硬件：

- 是

### 3.2 C2 通用水源控制器

用途：

- 机井主控
- 单泵泵站主控
- 小型泵站主控
- 单机一体控制器

主要负责：

- 启停泵
- 采集流量、压力、电量、运行状态
- 对接软启动器、接触器、变频器
- 管理本点位的运行会话

适用场景：

- `机井`
- `泵站`
- `控制点`

是否必须独立硬件：

- 是

### 3.3 C3 阀控器

用途：

- 控制单个电磁阀 / 闸阀 / 单个出水口支路
- 作为最小阀控单元独立部署

主要负责：

- 单阀开关控制
- 阀位或到位反馈
- 可选挂 1 路流量或压力

适用场景：

- `控制点`
- `出水口`
- `单支路阀位`

是否必须独立硬件：

- 是

### 3.4 C4 采集监测控制器

用途：

- 纯采集型监测点
- 低控制、重采集的点位

主要负责：

- 采集压力、流量、液位、气象、电表、变压器状态
- 可选少量本地告警输出

适用场景：

- `监测点`
- `控制点`
- `泵站` 的补充采集侧

是否必须独立硬件：

- 不一定
- 可以和 `C2` 共平台，只是 IO 组合不同

### 3.5 C5 远程 IO / 扩展控制器

用途：

- 补充 DI / DO / AI / AO / RS485
- 解决一个主控不够带的情况

主要负责：

- 按地址挂到 `C1` 或 `C2`
- 被动执行 / 上送通道数据

适用场景：

- 多泵、多阀、大泵站
- 采集点数量较多
- 主控与现场设备分散

是否必须独立硬件：

- 是，但不应当作为平台顶层主控角色直接出现

## 4. 不建议单独立项成控制器角色的对象

这些在业务上有用，但不值得做成独立“控制器大类”：

### 4.1 变频器控制器

建议定位：

- `C2` 的受控外设
- 或 `C5` 下挂从设备

原因：

- 变频器是控制能力的一部分，不是新的站点控制中心

### 4.2 PLC / RTU Controller

建议定位：

- 硬件实现形态
- 不是对业务用户暴露的一级角色

原因：

- 用户关心的是“泵站主控”还是“站级网关”，不是底层是不是 PLC

### 4.3 操作端 / 从控屏 / 刷卡端

建议定位：

- `H5 操作面板`
- 作为 `C2` 或 `C1` 的附属终端

原因：

- 它不应当拥有独立站点控制语义
- 老平台里的 `MASTER / SLAVE / SINGLE` 更适合改成部署模式，而不是产品大类

## 5. 硬件主 SKU 规划

## 5.1 H1 站级边缘网关

推荐职责：

- 对上跑 `tcp-json-v1`
- 对下管理 RTU、阀控板、采集板
- 本地命令缓存、断线补发、日志缓存、OTA

建议接口：

- 4G：`1`
- 以太网：`1` 到 `2`
- RS485：`2` 到 `4`
- USB / 调试口：`1`
- DI / DO：各 `2` 路，维护级即可
- 本地存储：`eMMC` 或 `TF`

推荐场景：

- 泵站
- 水肥站
- 大型井群

## 5.2 H2 通用 RTU 控制器

这是最关键的一块主控板，建议优先做。

推荐职责：

- 水泵启停
- 接触器 / 软启动 / 变频联动
- 本地点位运行控制
- 本地采集和上报

建议接口：

- 4G 或以太网：`1`
- RS485：`2`
- DI：`6` 到 `8`
- DO：`4` 到 `8`
- AI：`4`
- 脉冲输入：`2`
- AO：可选 `1` 到 `2`

建议覆盖的业务角色：

- `机井主控`
- `泵站主控`
- `单机一体控制器`
- `小型采控一体监测控制器`

也就是说：

- 机井主控和泵站主控，不要做两种板
- 同一块 `H2`，通过固件模板和端子映射区分角色

## 5.3 H3 单路阀控终端

这是第二块建议优先做的主板。

推荐职责：

- 单阀控制
- 单出水口控制
- 单支路启闭

建议接口：

- 上行通信：RS485 优先，可选 4G/NB
- DO：`1` 到 `2`
- DI：`1` 到 `2`
- AI：可选 `1`
- 脉冲输入：可选 `1`

建议覆盖的业务角色：

- `阀控器`
- `出水口控制器`
- `单支路控制器`

## 5.4 H4 采集 / 远程 IO 扩展板

推荐职责：

- 采集密集型场景
- 主控 IO 扩容
- 远程设备接入

建议接口：

- RS485：`2`
- AI：`8` 到 `16`
- DI：`8`
- DO：`4`
- 脉冲输入：`2`

建议覆盖的业务角色：

- `采集监测控制器`
- `远程 IO 扩展控制器`

## 5.5 H5 操作面板 / HMI

这不是主控 SKU，只是可选附属件。

推荐职责：

- 本地操作
- 状态显示
- 刷卡 / 二维码 / 人机交互

上行方式：

- 串口或 CAN 挂到 `H2` / `H1`

## 6. 角色与 SKU 对应关系

| 业务角色 | 推荐硬件 SKU | 是否独立联网 | 说明 |
| --- | --- | --- | --- |
| 站级网关控制器 | `H1` | 是 | 站级主通信入口 |
| 机井主控 | `H2` | 是 | 不要单独再做“机井专用板” |
| 泵站主控 | `H2` 或 `H1+H2` | 是 | 小站用 `H2`，大站用 `H1+H2` |
| 单机一体控制器 | `H2` | 是 | 老平台 `SINGLE` 建议落到部署模式 |
| 阀控器 | `H3` | 可否 | 当前按单路部署，可直连，也可挂网关 |
| 采集监测控制器 | `H4` 或 `H2` | 可否 | 纯采集优先 `H4` |
| 远程 IO 扩展 | `H4` | 否 | 不做顶层业务主控 |
| 操作端 / HMI | `H5` | 否 | 附属面板，不是主控 |

## 7. 推荐的收敛方案

如果你现在就要做硬件规划，我建议收敛成下面这套：

### 第一阶段必须做

1. `H2 通用 RTU 控制器`
2. `H3 单路阀控终端`

因为只靠这两块，已经能覆盖：

- 机井
- 小型泵站
- 单机版
- 出水口
- 支路阀组

### 第二阶段建议补

3. `H1 站级边缘网关`
4. `H4 采集 / 远程 IO 扩展板`

这两块主要解决：

- 大泵站
- 多设备聚合
- 复杂站级扩展
- 恢复编排和边缘缓存

### 第三阶段按需补

5. `H5 操作面板 / HMI`

## 8. 我建议冻结的嵌入式接口口径

后面嵌入式接口我建议按这条线来做。

### 8.1 北向统一

所有独立联网主控统一对上：

- `tcp-json-v1`
- 字段主键：`imei`
- 角色标识：`controller_code`
- 业务会话：`session_ref`
- 命令幂等：`command_id`

### 8.2 南向统一

主控对下尽量只保留 3 种方式：

1. `DI / DO`
2. `AI / Pulse`
3. `RS485(Modbus RTU 优先)`

不要一开始就把下位协议做得太花。

### 8.3 通道模型统一

每个硬件通道都映射成标准类型：

- `relay_output`
- `dry_contact_input`
- `pulse_counter`
- `analog_input_4_20ma`
- `analog_input_0_10v`
- `analog_output`
- `rs485_slave`

这样上层平台就不用关心底层是哪种板。

## 9. 业务规划建议

如果站在你现在硬件立项的角度，我的建议是：

### 9.1 不要按业务名字立太多板卡

不要立这些独立 SKU：

- 机井主控板
- 泵站主控板
- 单机版主控板
- 压力采集板
- 流量采集板
- 液位采集板

这些会把 SKU 做炸。

### 9.2 应按“能力组合”立 SKU

建议只按能力组合立：

- `联网主控`
- `多路执行`
- `多路采集 / 扩展`
- `站级网关`

### 9.3 平台展示上仍然保留业务角色

虽然硬件收敛，但平台上仍要区分：

- 机井主控
- 泵站主控
- 阀控器
- 监测控制器
- 站级网关

这样业务语义清楚，硬件 SKU 也不会失控。

## 10. 最终建议

如果你现在问我“控制器需要几种类型”，我的建议是分两句话回答：

### 对业务说

控制器角色先冻结 `5` 类：

1. 站级网关控制器
2. 通用水源控制器
3. 阀控器
4. 采集监测控制器
5. 远程 IO / 扩展控制器

### 对硬件说

主 SKU 先做 `4` 类就够：

1. `H1 站级边缘网关`
2. `H2 通用 RTU 控制器`
3. `H3 单路阀控终端`
4. `H4 采集 / 远程 IO 扩展板`

其中第一批真正必须优先打样的，只有两块：

1. `H2`
2. `H3`

这套最顺，也最不容易后面推翻。

## 11. 当前单路阀控的收口判断

如果你们当前现场全部是单路阀控，那么这条线可以直接收成 `1` 类：

- 业务角色只保留一个：`阀控器`
- 硬件 SKU 只保留一个：`H3 单路阀控终端`

不需要再区分：

- 单路阀控器
- 多路阀控器
- 出水口专用阀控器
- 支路专用阀控器

平台层统一看成一个 `valve_controller` 即可，区别只放在：

- 安装点位
- 绑定阀门
- 通道编号

但这只适用于“阀控”这条线。

整个控制器体系我仍然不建议缩成一类，因为：

- 水源主控和阀控的电气边界不同
- 运行会话责任不同
- 采集密度和供电边界也不同

所以我的最终建议是：

- `阀控器` 可以缩成 `1` 类
- `控制器总体系` 仍建议至少保留 `3` 类：
  - `站级网关控制器`
  - `通用水源控制器`
  - `阀控器`

如果当前阶段还不做站级网关，那么第一批甚至可以先只做 `2` 类：

- `通用水源控制器`
- `阀控器`
