# 嵌入式控制器 4G 联网与发包规则 v1

## 1. 目标

这份文档用于冻结 4G 联网与发包层的实现规则，避免设备在现场因为：

- 断网重连混乱
- 发包节奏过密
- 心跳与快照重复
- ACK 丢失后重复执行
- 缓存补发无边界

导致不稳定。

## 2. 传输层规则

- 传输方式：`TCP`
- 帧格式：`4-byte big-endian length + UTF-8 JSON`
- 单帧建议上限：`2048 bytes`
- 超过 `2048 bytes` 的状态快照，必须分拆或裁剪
- 不允许单帧塞入大量非必要字段

## 3. 连接状态机

建议固定为：

- `MODEM_BOOT`
- `NET_ATTACHING`
- `TCP_CONNECTING`
- `TCP_ONLINE`
- `TCP_DEGRADED`
- `TCP_BACKOFF`

## 4. 发包分类

### 4.1 必发包

- `RG`
  首次上线或重连成功后发
- `HB`
  轻量心跳
- `AK/NK`
  命令闭环必发

### 4.2 条件发包

- `SS`
  状态显著变化时发
- `ER`
  事件触发时发
- `QS`
  收到查询后返回

### 4.3 慎发包

- 大量全量快照
- 高频重复状态
- 对平台没意义的调试数据

## 5. 周期建议

| 报文 | 周期/触发 |
| --- | --- |
| `HB` | `30s` 默认 |
| `SS` | `60s` 默认，或状态变化时立即发送 |
| `ER workflow_runtime_tick` | `15s / 30s / 60s`，按业务场景配置 |
| `RG` | 上线后立即发送一次 |

## 6. 发包裁剪规则

### 6.1 `HB`

只带：

- `rd`
- `wf`
- `cv`
- `csq`
- `bs`

不要在心跳里带：

- 全量通道值
- 全量模块状态
- 大量历史字段

### 6.2 `SS`

只带当前最关键快照：

- workflow 状态
- 本次会话的累计量
- 必要通道值
- 必要故障码

不要带：

- 未启用模块
- 未变化通道
- 平台可以推导出的冗余字段

### 6.3 `ER`

事件包只带：

- 事件类型
- 关键结果
- 关联会话
- 关键累计量

## 7. 命令下发与回执规则

### 7.1 ACK

收到可执行命令后，先回：

- `AK`

仅表示：

- 已接单
- 进入执行

### 7.2 NACK

前置条件不满足时回：

- `NK`

必须带：

- `c`
- `rc`

### 7.3 不重复执行原则

若收到重复命令：

- 优先按 `c` 去重
- 若已执行，返回同一关联结果或幂等 ACK
- 不允许因 4G 抖动导致重复开泵、重复开阀

## 8. 缓存与补发

### 8.1 必须缓存的包

- `AK`
- `NK`
- `ER`
- 参与结算的 `workflow_runtime_tick`
- `workflow_stopped`

### 8.2 可以丢弃的包

- 旧心跳
- 被更新快照覆盖的普通状态快照

### 8.3 补发顺序

按以下优先级：

1. 命令回执
2. 结算事件
3. 运行 tick
4. 状态快照
5. 心跳

## 9. 重连规则

- TCP 断开后进入退避重连
- 退避建议：`5s -> 10s -> 20s -> 30s -> 60s`
- 重连成功后：
  1. 先发 `RG`
  2. 再发最新 `HB`
  3. 再按优先级补发关键缓存

## 10. 稳定性约束

- 不允许用动态拼接大字符串反复 malloc
- JSON 组包优先使用固定缓冲区
- 浮点字段尽量限制精度
- 可枚举字段尽量走短码字典
- 调试日志与主协议发包必须解耦

## 11. AI 实现输入要求

任务书里若要求 AI 写 4G 通信与协议代码，必须明确：

- 心跳周期
- 快照周期
- tick 周期
- 最大包长
- 缓存队列长度
- 重连退避策略
- 幂等规则

否则 AI 实现很容易出现：

- 心跳洪泛
- 重发失控
- 包体过大
- 命令重复执行
