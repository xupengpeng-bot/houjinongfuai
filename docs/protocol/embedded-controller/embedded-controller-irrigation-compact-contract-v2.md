# 嵌入式灌溉控制器协议与短指令合同 v2

更新时间：2026-04-11

## 1. 这份文档给谁

只给嵌入式开发、固件开发、联调工程师。

这份文档只讲设备要理解的内容：

- 收什么命令
- 回什么 ACK/NACK
- 上报什么运行事实
- 上报什么计量值
- 遇到异常怎么处理

不要从这份文档推导平台订单逻辑、退款逻辑、计费逻辑。

## 2. 唯一结论

嵌入式当前只需要实现一套“紧凑线协议”：

- 顶层短字段
- 短消息类型
- 短查询码
- 短动作码

平台内部可能还会看到长名字，例如：

- `EXECUTE_ACTION`
- `QUERY`
- `pause_session`
- `start_pump`

但设备侧只需要认本文档列出的短码。

## 3. 顶层报文

统一使用：

```json
{
  "v": 1,
  "t": "EX",
  "i": "864869000000001",
  "m": "000321",
  "s": 321,
  "c": "cmd-123",
  "r": "S-6601D010",
  "p": {}
}
```

字段含义：

- `v`：协议版本，固定 `1`
- `t`：消息类型短码
- `i`：设备 IMEI
- `m`：消息 ID
- `s`：本地递增序号
- `c`：命令关联号，等于平台下发的 `command_token`
- `r`：会话号，可空
- `p`：业务体

## 4. 消息类型短码

设备当前只需要理解这些：

- `RG`：注册
- `HB`：心跳
- `SS`：状态快照
- `ER`：事件上报
- `QR`：查询
- `QS`：查询结果
- `EX`：执行动作
- `AK`：命令成功
- `NK`：命令失败
- `SC`：配置同步

## 5. 当前必须支持的查询短码

- `qcs`：查询公共状态
- `qwf`：查询工作流状态
- `qem`：查询电表累计量

设备收到 `QR` 后：

- 能返回结果就回 `QS`
- 不能执行就回 `NK`

## 6. 当前必须支持的动作短码

### 当前正式动作短码

- `spu`：启动水泵
- `tpu`：停止水泵
- `ovl`：打开阀门
- `cvl`：关闭阀门
- `pas`：暂停当前灌溉
- `res`：恢复当前灌溉

### 当前不要求设备理解的平台词

以下词可能出现在平台代码或说明中，但不是设备要直接理解的线码：

- `start_session`
- `stop_session`
- `pause_session`
- `resume_session`
- `start_pump`
- `stop_pump`
- `open_valve`
- `close_valve`

设备侧只认短码：

- `spu`
- `tpu`
- `ovl`
- `cvl`
- `pas`
- `res`

## 7. ACK / NACK 规则

### ACK 的唯一含义

`AK` 只能表示：

- 动作已经真正成功

不能表示：

- 只是收到命令
- 只是开始执行
- 只是进入排队

### NACK 的唯一含义

`NK` 表示：

- 动作没有成功

常见原因：

- 设备忙
- 当前状态不允许
- 模块未启用
- 参数非法
- 安全联锁阻止
- 电源条件不满足

## 8. 各动作执行规则

### `spu`

- 收到 `spu`
- 真正启动成功后回 `AK`
- 启动失败回 `NK`
- 成功后尽快上报 `SS` 或 `ER`
- 上报后工作流状态必须进入运行态

### `tpu`

- 收到 `tpu`
- 真正停机成功后回 `AK`
- 停机失败回 `NK`
- 成功后必须上报最终停机事实
- 必须带最终累计计量

### `ovl`

- 收到 `ovl`
- 阀门真正打开后回 `AK`
- 打开失败回 `NK`

### `cvl`

- 收到 `cvl`
- 阀门真正关闭后回 `AK`
- 关闭失败回 `NK`

### `pas`

- 收到 `pas`
- 真正暂停成功后回 `AK`
- 暂停失败回 `NK`
- 暂停成功后运行秒数停止增长

### `res`

- 收到 `res`
- 真正恢复成功后回 `AK`
- 恢复失败回 `NK`
- 恢复失败后设备保持暂停态

## 9. 设备本地状态机

设备必须实现：

```text
READY -> STARTING -> RUNNING -> PAUSING -> PAUSED -> RESUMING -> RUNNING -> STOPPING -> STOPPED
READY -> STARTING -> FAULT
RUNNING -> FAULT
PAUSING -> FAULT
PAUSED -> FAULT
RESUMING -> FAULT
STOPPING -> FAULT
FAULT -> STOPPED
```

## 10. 工作流状态短码

设备在 `HB` / `SS` / `ER` 中统一上报 `wf`：

- `RI`：空闲可启动
- `ST`：启动中
- `RN`：运行中
- `PA`：暂停中
- `PS`：已暂停
- `RS`：恢复中
- `SP`：停机中
- `ED`：已停机
- `ER`：异常停机

## 11. 设备必须持续上报的累计量

统一要求：

- `rt`：累计运行秒
- `fq`：累计水量
- `ek`：累计电量 kWh

必要时也可带：

- `ew`：累计电量 Wh
- `pw`：当前功率
- `vv`：当前电压
- `ia`：当前电流
- `fm`：瞬时流量
- `pr`：瞬时压力

## 12. 计量规则

### `rt`

- 只有真正运行时增长
- 暂停时不增长
- 停机时不增长

### `fq`

- 只在真实出水时增长

### `ek`

- 只在真实耗电时增长

## 13. 启动、暂停、恢复、停机后的必须上报

### 启动成功后

必须上报：

- `wf=RN`
- `rt`
- `fq`
- `ek`
- `r`
- `c`

### 暂停成功后

必须上报：

- `wf=PS`
- `rt`
- `fq`
- `ek`
- `r`
- `c`

### 恢复成功后

必须上报：

- `wf=RN`
- `rt`
- `fq`
- `ek`
- `r`
- `c`

### 停机成功后

必须上报：

- `wf=ED`
- `rt`
- `fq`
- `ek`
- `r`
- `c`
- 停机原因

## 14. 异常停机

以下情况必须主动上报异常停机：

- 断电
- 故障
- 急停
- 接触器跳闸
- 保护停机

上报要求：

- `wf=ER` 或最终 `wf=ED`
- 带最终 `rt`
- 带最终 `fq`
- 带最终 `ek`
- 带异常原因

## 15. 刷卡事件

每次刷卡必须单独上报事件，不要和执行结果混在一起。

要求：

- 一次刷卡，一条事件
- 同卡同设备再次刷卡，设备优先按“结束当前灌溉”理解
- 去抖只能做极短窗口
- 不能吞掉第二次真实刷卡

## 16. 重连和重启

### 可以做的

- 补发历史事实事件
- 补发最后一次状态快照

### 禁止做的

- 重连后自动重发旧 `spu`
- 重启后自动补执行旧启动

## 17. 计数器重置

发生以下情况必须显式上报：

- 设备重启
- 计数器清零
- 换表
- 采集模块重置

至少要有一种机制：

- `meter_epoch`
- `counter_reset` 事件

## 18. NACK 推荐拒绝码

推荐设备侧至少支持：

- `BZ`：设备忙
- `UC`：不支持的命令
- `MN`：模块未启用
- `SI`：安全联锁
- `PR`：电源未就绪
- `PI`：参数非法
- `EX`：命令已过期

## 22. 当前唯一要求

嵌入式联调时，只按本文档实现。

不要再参考旧的动作短码写法，例如：

- `st`
- `sp`
- `ps`
- `rs`
- `ov`
- `cv`
- `sv`
- `tv`

这些旧短码在当前扫描灌溉控制器链路中不作为正式联调短码使用。
