## 3. 目标

补齐维护团队最小后端基线，使前端可以在稳定 contract 上落 `Project` 和 `Asset` 的团队挂载。

## 4. 范围

- 文档
- DDL / migration
- seed 样例
- 最小 REST contract
- 测试骨架或最小 smoke
- `docs/uat/lovable-codex-sync.md` 更新

不允许做：

- 成员管理
- 排班
- 工单自动派发
- 复杂组织树

## 5. 冻结模型

### maintenance_team

最小字段：

- `id`
- `team_name`
- `leader_name`
- `contact_phone`
- `status`
- `remarks`

### 关系

- `project.maintenance_team_id`
- `asset.maintenance_team_id`

### 生效规则

- Asset 有自己的团队时，优先 Asset
- Asset 没有自己的团队时，继承 Project 默认团队

## 6. 期望交付

1. migration
2. 最小 seed
3. `maintenance-team` list / options / create / update 的后端 contract
4. `project` / `asset` 读模型补齐：
   - 原始挂载字段
   - 实际生效团队展示字段
5. integration 或 e2e skeleton
6. 把结果回写 `docs/uat/lovable-codex-sync.md`

## 7. 验收口径

- 前端能够消费团队 options
- Project 能配置默认团队
- Asset 能配置覆盖团队
- Asset 详情能读到“实际生效团队”
- 不引入第二套冲突语义
