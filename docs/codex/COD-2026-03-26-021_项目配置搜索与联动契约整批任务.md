# COD-2026-03-26-021 项目配置搜索与联动契约整批任务

Status: closed
Audience: Cursor
Purpose: deliver the next larger-granularity backend wave for project-configuration linkage and searchable options so the frontend can close interaction quality in one larger pass.

## 1. 当前阶段

- Phase 1
- 项目配置真实交互深化阶段
- 当前目标：一次补齐区块/计量点相关的搜索型 options 和项目联动 contract

## 2. 任务目标

整批收口以下后端能力：

1. `project-blocks/options`
   - 支持按 `project_id` 过滤
   - 支持 `q` 搜索
2. `metering-points/form-options`
   - 支持按 `project_id` 返回收窄后的 `blocks / assets / devices`
   - 保持 `point_types / statuses`
3. 统一搜索型 options 契约
   - `projects`
   - `blocks`
   - `assets`
   - `devices`
   - 返回统一 `value / label` 结构
4. 为前端大列表搜索选择器提供最小可用后端支撑

## 3. 范围

允许做：

- controller / service / dto
- options / form-options query params
- response shape normalization
- build / test
- docs / RESULT 回写

不允许做：

- 修改前端业务代码
- 重开架构
- 扩到 DWG / solver / 成本分摊高级规则

## 4. 具体要求

### 4.1 `GET /api/v1/project-blocks/options`

新增支持：
- `project_id`
- `q`

要求：
- `project_id` 存在时，仅返回该项目下区块
- `q` 存在时，支持按编码 / 名称搜索

### 4.2 `GET /api/v1/metering-points/form-options`

新增支持：
- `project_id`
- `q`（至少对 blocks / assets / devices 生效，若实现成本过高可在本轮只做 `project_id`，但需明确说明）

要求：
- 选中项目后，区块候选必须可按项目收窄
- 资产 / 设备候选优先也随项目收窄
- 继续返回 `point_types` / `statuses`

### 4.3 统一响应壳

对 options 类接口，尽量统一为：
- `value`
- `label`
- 若前端联动需要，可附加最小元数据，如 `project_id`

## 5. 验收口径

1. 后端提供项目 -> 区块联动的正式 contract
2. 后端提供大对象搜索型 options 的最小支撑
3. build 通过
4. 明确说明：前端是否已具备进入下一批“大颗粒交互完善”条件
