# 事故复盘：2026-03-26 派单入口缺失

状态：active  
级别：流程事故  
范围：PM 派单 / Cursor 执行入口

## 1. 事故现象

在 `LVB-4021` 完成后，后续需要 Cursor 进行本地验收。  
但聊天里已经说了“让 Cursor 执行”，而 `docs/codex/CURRENT.md` 仍然保持：

- `active task = none`
- `Execute now = no automatic execution item is queued`

Cursor 按规则重读文件后拒绝继续执行，导致一次无效往返。

## 2. 根本原因

根本原因不是执行方不配合，而是 **PM 派单不完整**：

1. PM 在聊天层已经完成了切换判断
2. 但没有先把正式入口文件写完整
3. 于是出现“口头派单”和“文件真相”不一致

本质问题是：

**派单动作不是原子完成的。**

## 3. 负责人

**主责：PM / 调度层，也就是我这边。**

原因：

- Cursor 按文件执行，没有越权
- Lovable 之前多次按文件拒绝旧任务，也是在正确执行规则
- 这次是 PM 没先完成入口文件更新，就提前在聊天里下达执行指令

所以如果用户觉得 Cursor 和 Lovable 这次比 PM 更靠谱，这个判断在本次事件里成立。

## 4. 形成机制

这类事故发生的机制是：

1. PM 先在脑中完成队列切换
2. PM 先在聊天里告诉用户“让某执行主体执行”
3. 但 `CURRENT.md` / `WAVE.md` 尚未同步更新
4. 执行方按文件重读后发现仍是 waiting 或 none
5. 最终形成“说了执行，但文件没派单”的冲突

## 5. 永久预防规则

### 规则 1：无入口，不派单

如果 `CURRENT.md` 没有写明：

1. active task
2. work mode
3. execute-now instruction

则 PM 不得在聊天里说“让 Cursor 执行”或“让 Lovable 执行”。

### 规则 2：文件先于聊天

正式派单顺序必须是：

1. 更新任务文件
2. 更新 `CURRENT.md`
3. 更新 `WAVE.md` 或波次文件（如需要）
4. 再在聊天里只说“让某执行主体执行”

### 规则 3：执行方拒绝口头派单是正确行为

若执行方读到：

- `active task = none`
- `Status = waiting`
- `Execute now = do not execute`

则拒绝执行是正确行为，不应被视为执行方失误。

### 规则 4：聊天与文件冲突时，按 PM 派单失败处理

以后遇到“聊天说执行、文件没更新”的情况，一律按：

**PM 派单失败，必须先补正式入口，再重新派单。**

## 6. 结论

这次事故的根本原因是：

**PM 没有先完成 `CURRENT.md` 派单落盘，就提前在聊天里派单。**

这次事故的负责人是：

**PM / 调度层，也就是我这边。**
