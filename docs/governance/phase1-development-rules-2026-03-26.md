1. 任务理解
本文件是 2026-03-26 起本仓库 Phase 1 的统一开发约束与开发规范。

2. 全局影响检查
所有协作者必须遵守以下总原则：
- backend 是运行、计费、安全、权限、审计的唯一真相
- project 仍是经营与核算主体
- 数据范围骨架固定为 business_region -> project -> site_subtree
- 前端不得直接读取业务表，不得在本地推导核心业务规则
- 所有正式接口继续保持 /api/v1 与统一 envelope

3. 本次范围内
开发约束：
- 先主链路，后体验优化
- 先真实联调，后视觉细化
- 先测试与迁移留痕，后继续扩面
- 所有新任务先落 docs，再实施，再复审
- 所有评审必须写明基线：本地工作树、当前分支、还是 origin/main

4. 本次范围外
禁止事项：
- 擅自扩展 Phase 2 / 3
- 擅自修改冻结 contract 语义
- 擅自让前端承担计费、权限、运行、安全真相
- 擅自引入长期兼容层、双口径、脏映射

5. 实施方案
开发规范：
- 后端交付至少具备：代码、测试或迁移、接口口径说明
- 前端交付至少具备：代码、构建通过、真实请求路径说明、验证方式
- Review 结论后必须附带下一步可直接转发的派单文本
- 默认按大任务包并行推进，减少碎片化来回

6. 风险与依赖
风险：
- 若继续双真相或双主线推进，会快速放大沟通与返工成本
- 若不记录任务与评审基线，复审结果会失真
依赖：
- docs/p1、docs/uat、docs/protocol 作为冻结边界依据

7. 验收方式
验收标准：
- 以后所有任务按本规则执行
- Lovable / Cursor / Codex 的派单与复审口径一致
- 不再出现无留痕派单和无基线复审

8. 需要交接给谁
本文件作为统一治理规则长期生效，无需额外交接。
