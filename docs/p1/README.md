## Phase 1 固定原则

- 固定规则优先级：井级策略 > 关系配置 > 交互策略 > 模板 > 设备类型默认值。
- 所有运行、计费、安全、并发、停机保护的最终裁决统一由后端输出。
- 前端只负责发起受控动作、展示状态、展示阻断原因和展示后端返回的可执行动作。
- AI 服务仅预留会话中台、知识检索入口、转人工/转工单能力，不在 Phase 1 实现多渠道正式运营。
- Phase 1 的唯一目标是打通“对象配置 -> 规则生效 -> 扫码启动 -> 运行会话 -> 订单形成 -> 告警处理 -> 工单闭环 -> UAT 验收”。

## 推荐使用方式

- 后端研发：先读对象模型、状态机、表结构，再开始工程骨架和迁移。
- 前端研发：先读 API 草案，严格以 `decision`、`blocking_reasons`、`available_actions` 为显示来源，不自行推导业务结论。
- Lovable：只消费本目录中的模块边界、页面动作和对象字段，不扩新增一级菜单。
- UAT：按状态机和 API 的返回口径设计角色验收脚本。
