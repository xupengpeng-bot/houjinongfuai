# Phase 1 系统可用性收口需求包 v1

状态：`active`  
适用范围：后稷农服灌溉 AI 运营系统 `Phase 1`

## 1. 目标

当前阶段不再以“页面数量”作为进度核心，而是以“角色主路径是否顺畅、配置到调度到执行是否连贯、后端是否能支撑真实场景”作为收口标准。

本需求包的目标是把系统收成一套真正顺手的工作系统：

1. 管理端围绕两条主路径工作：
   - `配置工作台`
   - `调度工作台`
2. 移动运维端围绕一条任务链工作：
   - `待办 -> 工单 -> 现场处理`
3. 农户端围绕一条用水链工作：
   - `首页/扫码 -> 当前会话 -> 历史/帮助`
4. 设备与嵌入式围绕一条后端真相链工作：
   - `后端求解/命令注册表 -> 设备拉命令 -> ACK -> 运行/结算`

## 2. 设计原则

1. 后端是唯一业务真相，前端不再承担设备模拟器、求解器或业务规则计算职责。
2. 操作入口按“角色任务”组织，不按“系统模块”堆菜单。
3. 详情页是辅入口，不是主路径；主路径必须能在少跳转下完成任务。
4. 配置和调度必须分离：
   - 配置页负责项目、区块、地图 / DWG、关系、版本发布
   - 调度页负责运行摘要、求解预览、设备合同、最近回执
5. 状态迁移要贴近真实场景：
   - 例如停机不能再“点按钮就立刻结束”，而应由设备 ACK 驱动完成

## 3. 角色主路径

### 3.1 管理端 `ops`

推荐主路径：

1. `运营总览`
2. `配置工作台`
3. `调度工作台`
4. `运行监控 / 预警中心 / 工单中心`

详细页只作为补充：

- `模型详情`
- `设备关系`
- `泵阀关系`
- `日志审计`
- `UAT`

### 3.2 移动运维端 `m`

推荐主路径：

1. `我的待办`
2. `我的工单`
3. `现场处理`

关键规则：

- 现场处理必须按工单 ID 连续处理
- 不能再自动挑选“第一张可处理工单”
- 用户从哪张工单点进来，就应处理哪张工单

### 3.3 农户端 `u`

推荐主路径：

1. `首页`
2. `扫码`
3. `当前会话`
4. `历史 / 帮助`

关键规则：

- 结束会话后，先确认结果，再决定是否去历史或再次扫码
- 帮助页只放帮助，不混入记录与导航替代

### 3.4 设备 / 嵌入式

推荐主路径：

1. 后端生成调度与命令
2. 设备按 IMEI 拉取待执行命令
3. 设备回传 `COMMAND_ACK / RUNTIME_STOPPED`
4. 后端更新运行态、命令态、结算态

关键规则：

- 设备只对接后端合同
- 前端只展示后端结果

## 4. 当前主要问题

1. 管理端页面过多，主链路被拆散，用户要自己判断先后顺序。
2. 配置和调度虽然已经分出两页，但仍有细节页分散在外层导航中，容易打断操作节奏。
3. 移动运维端此前存在“点开的工单”和“现场处理实际处理的工单”不一致的问题。
4. 多处历史页面与布局存在文案和术语不统一问题，影响理解成本。
5. 监控、告警、处置类页面仍有重叠，需要继续收成更清晰的处置主线。

## 5. 当前已落地的收口动作

1. 管理端导航已改成以“系统主线、基础配置、网络与关系、运行与履约、支持与扩展”组织。
2. 顶部入口已突出：
   - `配置工作台`
   - `调度工作台`
   - `运维端`
   - `农户端`
3. `Dashboard` 已收成 `运营总览`，并提供三类主入口：
   - 配置工作台
   - 调度工作台
   - 处置工作台
4. 移动运维端 `MyTodos / MyWorkOrders / FieldProcess` 已改为按工单 ID 连续处理。
5. `stop session` 已收成 ACK 驱动完成链，运行与结算更贴近真实设备行为。

## 6. 当前优先级

### P0

1. 继续把 `配置工作台` 收成真正的一页主路径：
   - 项目 / 区块
   - 图源上传与分析
   - 关系自动生成
   - 草稿 / 发布
   - 去调度
2. 继续把 `调度工作台` 收成真正的执行主路径：
   - 调度就绪度
   - 后端求解预览
   - 设备合同
   - 最近回执
   - 去运行 / 去处置

### P1

1. 把监控、告警、工单进一步压成统一的处置主线。
2. 把农户端结束会话后的结果确认链再收顺。
3. 清理剩余历史脏文案和不一致术语。

### P2

1. 求解器从确定性编排继续向真实优化推进。
2. 真实硬件 transport / ACK 通道继续增强。

## 7. 验收标准

满足以下条件，才认为“系统开发完善可用”进入稳定推进状态：

1. 管理端用户能够不查文档，仅凭页面引导完成：
   - 配置
   - 发布
   - 去调度
2. 移动运维端用户能够从待办直接进入正确工单并完成处理。
3. 农户端用户能够顺畅完成：
   - 扫码
   - 开始用水
   - 停止用水
   - 查看结果
4. 设备侧能够只依据后端合同完成：
   - 拉命令
   - 回 ACK
   - 驱动运行与结算闭环
5. 前端 `build`、关键 real-mode `smoke`、后端 `build` 持续通过。

## 8. 推荐推进顺序

默认按下面顺序持续推进：

1. `配置工作台` 收口
2. `调度工作台` 收口
3. `处置工作台` 收口
4. `农户端` 结果确认与帮助链优化
5. `求解器 / 硬件 transport` 深化
