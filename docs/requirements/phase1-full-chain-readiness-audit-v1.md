## 1. 审计目标

本审计不再只看“页面有没有开发”，而是看系统是否已经具备下面 3 个特征：

1. 高效
   - 用户不需要理解内部模块边界，就能完成核心任务。
2. 稳定
   - 主路径持续可跑，构建、烟测、关键回执链不容易因为小改动断掉。
3. 准确
   - 前端不伪造设备与调度真相，后端、设备、结算的状态迁移符合真实场景。

## 2. 当前结论

当前系统已经从“功能页集合”进入“主链可运行”的阶段，但距离“全流程高效稳定准确跑起来”还差最后一层收口。

当前判断：

- 主链可运行度：`75% - 80%`
- 可持续研发稳定度：`65% - 70%`
- 真正达到“高效稳定准确”的就绪度：`60% - 65%`

## 3. 已经成立的能力

### 3.1 配置到调度主链已成型

当前已经具备：

- `配置工作台`
  - 项目/区块选择
  - 图源上传
  - 图源分析
  - 草稿/发布
  - 后端自动生成设备联动关系
- `调度工作台`
  - 后端交接包
  - 后端调度参数
  - 后端设备协议合同
  - 最近设备回执

### 3.2 运行与设备主链已成型

当前已经具备：

- 后端统一命令注册表
- 设备按 `IMEI` 拉待执行命令
- `HTTP + TCP socket` 双通道回执
- `ACK` 驱动运行状态更新
- `stop session` 不再立即结算，而是等待设备确认

### 3.3 质量闸门已有基础

当前已验证通过：

- `backend npm run build`
- `backend npm run test:unit`
- `frontend npm run build`
- `frontend npm run test:smoke`

## 4. 当前关键缺口

### P0. 主路径仍未完全收成“一页配置，一页调度，一条处置”

当前虽然已经有：

- `配置工作台`
- `调度工作台`

但仍存在以下割裂点：

- 配置动作仍会跳到 `模型详情`、`设备关系` 等详情页完成理解
- 监控、告警、工单、运行容器、审计之间仍然是多个入口并存
- 还没有真正收成一条“处置工作台”主线

影响：

- 用户需要自己判断下一步去哪
- 系统会显得功能很多，但主路径不够短

### P0. 调度求解器还不是最终态

当前求解器已经能输出可执行步骤和设备合同，但本质仍是：

- 确定性编排
- 可执行预览

还不是：

- 真实优化求解
- 多目标权衡
- 可比较的求解方案

影响：

- 当前足够支撑前后端与嵌入式联调
- 但还不足以称为最终调度内核

### P0. 真实硬件接入仍停在“后端已就绪”

当前设备链已经具备：

- 后端命令队列
- `HTTP ingest`
- `tcp-json-v1` socket
- ACK 关联

但还没有正式完成：

- 真实串口/网关桥接
- 超时重试
- 死信/补偿
- 设备断连后的恢复策略

影响：

- 本地模拟与联调可做
- 真设备大规模稳定接入还不能算闭环

### P1. 农户端和移动端还有“结果态体验”缺口

当前主链能跑，但还差最后一层用户体验收口：

- 农户端停止用水后的结果确认还不够强
- 农户端“因已有运行会话而不能开新会话”虽然后端会拒绝，但前端还可以更清晰解释原因与下一步
- 移动运维端虽然已经按工单 ID 连续处理，但“处置完成后的回流”和“下一步建议”还可以更顺

### P1. 原始接口合同还不够统一

当前后端整体已经能支撑前端和设备，但仍能看到一些原始合同层问题：

- 有的接口是 `{ requestId, code, message, data }`
- 有的前端服务再额外做一次 `res.data ?? res`
- 有的字段使用 `queuedCommands`
- 其他地方仍有 `snake_case` 语义

影响：

- 页面上未必立刻出错
- 但长期会增加前后端、嵌入式和测试联调成本

### P1. 前端工程稳定性还缺一道硬闸

当前前端：

- `build` 通过
- `smoke` 通过

但：

- `npm run lint` 失败
- 当前共有 `120` 个 error、`18` 个 warning
- 主要集中在 `any` 泛滥、Hook 依赖、空接口、旧组件问题

这说明：

- 系统“能跑”
- 但工程层“还不稳”

### P2. 前端包体偏大

当前构建输出：

- 主包约 `826 kB`

这不是阻塞问题，但会影响：

- 后续加载效率
- 大屏或弱网络设备体验

## 5. 这次新确认并已修复的问题

本次审计中确认并修复：

1. `配置工作台` 在“只有 1 个项目 + 1 个区块”时没有自动选中，导致上传主路径被锁死。
2. `调度工作台` 在同样场景下也没有默认选中唯一范围。

现在两页都已经改成：

- 单项目单区块时自动选中
- 主路径打开即可开始工作

## 6. 建议的收口顺序

### 第一优先级

1. 把 `配置工作台` 继续压成真正的一页主路径
2. 把 `调度工作台` 继续压成真正的一页执行台
3. 把监控/告警/工单/运行容器收成一条 `处置工作台`

### 第二优先级

1. 统一前后端原始接口合同风格
2. 清理前端 lint 错误，建立稳定工程闸门
3. 收农户端与移动端结果态体验

### 第三优先级

1. 求解器从确定性编排升级到真实优化
2. 真实硬件 transport / ACK 通道增强
3. 包体拆分与性能优化

## 7. 验收标准

当下面这些同时成立，才算“全流程高效稳定准确跑起来”：

1. 管理端用户可以不查文档完成：
   - 配置
   - 发布
   - 调度
   - 处置
2. 移动运维端用户可以从待办进入正确工单并完成闭环
3. 农户端用户可以顺畅完成：
   - 启动前检查
   - 创建会话
   - 停止会话
   - 查看结果
4. 设备侧可以只依赖后端合同完成：
   - 拉命令
   - 回 ACK
   - 驱动运行与结算
5. 质量闸门持续为绿：
   - `backend build`
   - `backend unit test`
   - `frontend build`
   - `frontend smoke`
   - `frontend lint`
