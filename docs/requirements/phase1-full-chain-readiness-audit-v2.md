## 1. 审计目标

本审计关注的不是“页面有没有”，而是系统是否已经具备下面 3 个特征：

1. 高效：用户能沿主路径完成配置、调度、处置，不需要自己猜顺序。
2. 稳定：构建、单测、烟测和关键主链不会因局部改动轻易断掉。
3. 准确：调度、设备、结算、工单、告警都以后端真实状态为准。

## 2. 当前结论

当前系统已经进入“主链可运行”阶段，但还没达到“全流程高效、稳定、准确跑起来”的最终态。

- 主链可运行度：`80%`
- 研发稳定度：`70%`
- 整体就绪度：`65% - 70%`

## 4. 当前关键缺口

### P0. 求解器还不是真正优化内核

当前 solver 已能输出：
- 可执行步骤
- 设备脚本
- 嵌入式接口合同

但本质仍然是：
- 确定性编排预览
- 可执行预览

还不是：
- 真正优化求解
- 多目标权衡
- 可比较候选方案

### P0. 真实硬件 transport 还没到生产级

当前已经做到：
- 命令下发
- 设备拉取
- ACK / NACK 回写
- retry / backoff / dead letter
- TCP socket ready
- 连接健康可视化
- 后台自动 sweep

但还缺：
- 串口 bridge
- 外部网关 bridge
- 自动断连恢复编排
- 生产级重试 / 恢复策略

### P0. 处置工作台还没完全压平

`/ops/disposal` 已经能做同页处理，但深层细节仍散在：
- `/ops/run-monitor`
- `/ops/alert-center`
- `/ops/work-orders`
- `/ops/audit`

### P1. 农户端与移动端结果态还可继续收

重点问题：
1. 停止后的确认页还可以再补更多结果解释。
2. 完成处理后的附件、拍照等深层动作仍是占位入口。
3. 农户端与移动端的结果术语还可以再统一。

### P1. 接口合同风格还没统一

当前仍存在：
- `envelope` 风格不完全一致
- `snake_case` 与 `camelCase` 并存
- 前端部分服务还要做兜底适配

### P1. 前端工程闸门还没闭环

当前状态：
- `frontend build` 通过
- `frontend smoke` 通过
- `frontend lint` 失败

已知状态仍然是：
- `118 errors / 21 warnings`

## 6. 推荐收口顺序

### 第一优先级

1. 继续把处置工作台压成真正的一条连续操作流。
2. 继续往真实硬件 `transport / bridge` 推进。
3. 继续把求解器从编排预览推进到真实优化内核。

### 第二优先级

1. 再收一轮农户端和移动端结果态。
2. 统一前后端原始接口风格。
3. 清前端 `lint`。

### 第三优先级

1. 包体优化。
2. 更细的角色权限、异常态和导出能力收口。

## 7. 验收标准

只有当下面这些同时成立，才算接近“全流程高效稳定准确跑起来”：

1. 配置人员可独立沿主路径完成图源分析、草稿保存、发布。
2. 调度人员能看懂求解结果、transport 风险和恢复建议。
3. 值守人员能在处置台完成绝大多数告警闭环。
4. 农户端用户能顺畅完成开始、停止、查看结果。
5. 移动运维能从待办进入工单并顺畅完成处理。
6. `backend build`、`backend unit test`、`frontend build`、`frontend smoke`、`frontend lint` 全绿。
