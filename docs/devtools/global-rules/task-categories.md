# 全局任务分类规则

Status: active  
Audience: PM, Cursor, Lovable  
Scope: 全局协作规则，不属于业务研发任务

## 1. 任务类别

以后正式任务只允许以下三类：

1. 业务研发任务
2. 验收协同任务
3. 开发工具相关任务

## 2. 定义

### 2.1 业务研发任务

定义：

- 直接推进产品业务功能
- 直接影响业务闭环、操作流程、真实用户体验

典型范围：

- 项目 / 区块 / 计量点
- 设备配置
- 设备联动
- 地图配置
- 求解器
- 驾驶舱
- 订单 / 会话 / 告警 / 工单
- 数据权限落到真实业务查询

### 2.2 验收协同任务

定义：

- 不新增业务能力
- 只用于同步、构建、验证、联调确认、结果回写

典型范围：

- handoff sync
- 本地 verify
- build / test
- GitHub main 确认
- 联调结论回写

### 2.3 开发工具相关任务

定义：

- 不直接增加业务能力
- 只服务研发协作效率、自动化、任务系统、工具链

典型范围：

- dispatch 数据库
- 任务状态机
- sequencing
- shared connection layer
- 自动推进
- 开发脚本与协作工具

## 3. 死规则

### 3.1 方案讨论不派单

以下场景默认不下发任务：

- 业务方案讨论
- 架构评审
- 流程梳理
- 逻辑分析
- 风险讨论

只有在用户明确表示“继续研发”“开始实现”“下发任务”“让 Cursor 执行”“让 Lovable 执行”时，才进入正式执行。

### 3.2 开发工具相关任务不抢主线

没有用户明确批准时：

- 不主动下发开发工具相关任务
- 不允许开发工具相关任务抢占业务研发主线

### 3.3 派单必须标明类别

以后每次正式派单，必须先明确写明：

- 任务类别：业务研发任务
- 或 任务类别：验收协同任务
- 或 任务类别：开发工具相关任务

没有类别的任务，不进入正式执行流。

## 4. 存放规则

### 4.1 业务相关资料

继续放在原业务目录，例如：

- `docs/p1/`
- `docs/uat/`
- `backend/src/modules/...`

### 4.2 开发工具相关资料

必须单独存放，不与业务资料混放，例如：

- `docs/devtools/`
- `backend/sql/dispatch-mysql/`
- `backend/tools/`
- `scripts/devtools/`

## 5. 追溯规则

本文件是当前全局任务分类规则真相源。

后续若数据库任务模式需要引用该规则：

- 优先引用本文件路径
- 不在热路径数据库字段中重复存长正文
- 数据库只存版本号、摘要和引用
