# 后稷农事服务AI综合平台：Codex / Lovable 启动提示词

## 1. 给 Codex 的第一条任务

```text
请基于《后稷农事服务AI综合平台》当前方案，先不要直接写完整前端页面，先输出研发基线，范围仅限 Phase 1：
- 角色与权限
- 区域管理
- 设备类型
- 设备台账
- 机井 / 泵 / 阀
- 计费包
- 井级策略
- 泵阀关系
- 扫码用水
- 会话
- 订单
- 告警
- 工单
- UAT
- AI 会话中台预留

请输出：
1. 核心对象模型文档
2. 状态机文档
3. API 草案文档
4. 数据表结构草案
5. 模块目录结构建议

要求：
- 规则优先级固定为：井级策略 > 关系配置 > 交互策略 > 模板 > 设备类型默认值
- 所有运行和计费判断统一由后端输出决策结果
- AI 服务仅预留会话中台，不在 Phase 1 直接实现多渠道正式运营
- 输出要可开发，不要泛泛总结
```

## 2. 给 Lovable 的第一条任务

```text
我要做一套农业灌溉与农事服务设备运营 SaaS 的 Phase 1 原型，只做 Phase 1，不要扩展到 3D、视频、复杂智能水网。

请生成三端原型：
1. /ops 管理后台
2. /m 运维移动端
3. /u 农户端

Phase 1 页面范围：
- 驾驶舱
- 区域管理
- 设备类型
- 设备台账
- 机井管理
- 计费包
- 井级策略
- 泵阀关系
- 订单列表/详情
- 告警列表/详情
- 工单列表/详情
- UAT 验收台
- 用户/角色/权限
- /m 我的待办、我的工单、我的设备、现场处理
- /u 扫码、当前会话、结束、历史、帮助
- 预留 AI 入口与 AI 会话页壳子

要求：
- 专业、克制、偏工业运营风格
- 统一列表页、详情页、表单页、移动任务页规范
- 先用 mock 数据
- 不要自动扩出多余一级菜单
- 不要自行实现复杂规则引擎与多渠道 AI 逻辑
```

## 3. AI 服务分期结论

- Phase 1：只预留 AI 会话中台与帮助入口
- Phase 2：先做 App 内 AI 对话 1.0
- Phase 3：接微信 + 飞书
- Phase 4：做主动提醒与更强的农事建议
